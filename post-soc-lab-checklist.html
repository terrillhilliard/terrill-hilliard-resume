Building a Home SOC Lab: A Practical Starting Checklist

What actually matters when you spin up Elastic SIEM and Kali Linux for the first time — and how to avoid building a lab that just sits there looking impressive.

Every home SOC lab starts the same way: a fresh Elastic stack, a Kali box, and a lot of enthusiasm. Most of them stall out at the same point too — right after the install finishes and before any real logs start flowing. The gap between "I set up a SIEM" and "I can detect something" is where most of the learning actually happens.

Here's the order that's worked well for getting a lab from zero to genuinely useful, rather than a demo that only proves the software installed correctly.

1. Start with one well-instrumented endpoint, not five

It's tempting to spin up a whole network on day one. Resist it. A single Windows or Linux VM with Elastic Agent installed, forwarding logs into Elasticsearch, teaches you more than five under-configured hosts. Get comfortable with:

  • What "normal" log volume looks like for that one host
  • Which data streams are actually populated (process, network, file, registry)
  • How long ingestion lag is under load

2. Ingest before you attack

A common mistake is running an attack from Kali before confirming the target's logs are actually landing in the SIEM. Generate a known, benign event first — a failed login, a new scheduled task — and confirm you can find it. If you can't find the boring event, you won't find the attack either.

3. Write detections against your own baseline

Out-of-the-box detection rules are a good starting point, but the real skill is writing one from scratch against a signal you generated yourself. A simple, high-value first rule: alert on a process spawning from an unusual parent (e.g., winword.exe spawning powershell.exe). It's a classic macro-payload pattern, it's easy to simulate safely, and it teaches you how to read process-tree data in your SIEM of choice.

4. Run the attack, then work backward

Once ingestion and a baseline detection are working, use Kali to run something simple and well-documented — a Metasploit reverse shell or a basic reconnaissance scan — against the instrumented host. Then go find it in the SIEM without looking at the attack console first. That's the actual exercise: reconstructing what happened from logs alone, the same way you would during a real investigation.

5. Keep a findings log

Every lab session should end with a short written note: what you tested, what you saw, what you missed, and what you'd tune next. This is the single highest-leverage habit in the whole exercise — it's what turns "I played with a SIEM" into a body of documented, explainable work you can talk through in an interview.

Where this leads

A home lab built this way scales naturally into more advanced scenarios — lateral movement detection, persistence mechanisms, log correlation across multiple hosts — without ever feeling like a random pile of tools. The goal isn't a bigger lab. It's a lab you can explain, end to end, in plain language.
